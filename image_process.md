# OCR 전처리: 기하학적 보정 (Geometric Correction)

> **[2026-07-21 업데이트] 이 프로젝트 문서 특성 반영**
> 우리 문서는 폰카메라로 촬영한 게 아니라 대부분 **HWP→PDF 변환본이거나, 개인정보를 마킹펜으로 수기 마스킹한 뒤 평판/시트피드 스캐너로 스캔**한 것으로 예상됨. 이 조건에서 아래 항목들의 실제 우선순위는:
> - **디스큐(1.1)**: 우선순위 높음 — 스캐너 급지 과정에서 종이가 살짝 비뚤어지는 경우가 흔함
> - **원근 보정(1.2)**: 우선순위 낮음 — 폰 촬영 문서(책 곡률, 키스톤 왜곡)에 최적화된 기법들이라, 평판 스캐너 스캔본에는 대부분 불필요. 다만 스캔 원본 자체가 아니라 "스캔본을 스마트폰으로 재촬영해서 보낸" 케이스가 섞여 있으면 이 섹션이 다시 중요해짐 (초반 샘플 확인 시 체크 필요)
> - **잡음/마스킹 영역 처리(1.4, 신규)**: 우선순위 높음 — 마킹펜 마스킹, 스캔 노이즈, 이면 인쇄 비침(bleed-through) 대응이 이 프로젝트에서 실질적으로 더 중요한 이슈

## 1.1 디스큐 (Deskew)
스캔 시 기울어진 각도를 감지해 회전 보정하는 작업.

### 방법 A: OpenCV 기반

**장점**
- OCR 없이 빠르게 동작
- 텍스트 유무 관계없이 대략적인 레이아웃(선, 표 등)만으로도 가능

**단점**
- 텍스트가 짧거나 이미지에 표/그림이 섞이면 부정확
- 회전각이 큰 경우(45도 이상) 오판정 가능

**세부 기법**
| 기법 | 설명 |
|---|---|
| Hough Line Transform | 직선 검출 → 가장 많이 검출된 직선들의 각도 평균으로 회전각 추정 |
| Minimum Area Rectangle | 텍스트 픽셀들을 하나의 덩어리로 보고 `cv2.minAreaRect()`로 최소 외접 사각형을 구해 그 기울기를 회전각으로 사용 |

### 방법 B: EasyOCR bounding box 기반

**장점**
- 실제 텍스트 라인 기준이라 정확도가 높음
- 표나 그림에 방해받지 않음
- 다단 레이아웃에서도 라인 단위로 강건함

**단점**
- OCR 모델 자체를 돌려야 해서 느림 (GPU 없으면 특히)
- 텍스트가 거의 없는 문서(표만 있는 경우 등)엔 부적합

### 방법 C: Projection Profile (수평 투영 프로파일) — [2026-07-21 추가]

**개요**
후보 각도(예: -5° ~ +5°, 0.1° 간격)로 이미지를 조금씩 회전시켜보면서, 각 회전본의 가로 방향 픽셀(글자) 분포를 행(row) 단위로 합산 → 그래프로 그리면 텍스트 줄이 수평에 맞을 때 "봉우리(peak)"가 가장 뾰족해짐. 이 뾰족함(분산)이 최대인 각도를 정답으로 채택.

**장점**
- 스캐너로 스캔된 정형 문서(글자가 가로로 규칙적으로 배열)에 특히 강함 — 우리 문서(공문서, 표 위주) 조건과 잘 맞음
- OpenCV 단독보다 표/선에 덜 민감하고, EasyOCR보다 훨씬 빠름 (딥러닝 불필요)
- 회전각이 작은(±10도 이내) 스캔 문서에 정확도 높음 — 실무에서 가장 널리 쓰이는 고전 기법 중 하나

**단점**
- 회전각이 크면(문서가 90도 가까이 돌아간 경우) 탐색 범위를 넓혀야 해서 느려짐
- 텍스트가 매우 적은 페이지(표지, 이미지 위주 페이지)엔 부정확

**참고**: `Radon Transform` 기반 방식도 원리는 동일 (다양한 각도로 투영 후 분산이 가장 큰 각도 탐색). scikit-image `skimage.transform.radon`으로 구현 가능. Projection Profile보다 이론적으로 더 정교하지만 계산량이 더 큼.

> **이 프로젝트 추천**: 방법 A(Hough/minAreaRect)로 1차 추정 → Projection Profile로 소각도(±5도) 미세 보정, 이렇게 2단계로 조합하면 속도와 정확도를 둘 다 잡을 수 있음. EasyOCR 기반(방법 B)은 이미 OCR 파이프라인에 EasyOCR을 쓰고 있으니 박스 좌표를 재활용해 "공짜로" 디스큐 각도까지 얻는 것도 가능 (추가 연산 비용 거의 없음).
>
> **[2026-07-22 정정]** 실제 구현·실측해보니 이 가정은 틀렸음 — 아래 "구현 및 실측 결과" 참고 (EasyOCR은 속도도 제일 느리고, 각도 신뢰도도 제일 낮음).

- `n_samples`는 메소드마다 세는 대상이 다름 — 절대 서로 비교(신뢰도 지표)하면 안 되고, 메소드 내부에서만 의미 있음:

| 메소드 | n_samples가 세는 것 | 문서 내용에 따라 값이 달라지는가 |
|---|---|---|
| A1 Hough | 아웃라이어 제거 후 남은 직선 개수 | O |
| A2 minAreaRect | dilate 마스크의 컨투어(텍스트 블록) 개수 | O |
| B1 EasyOCR | 최빈 각도 구간(dominant cluster)에 속한 박스 개수 | O |
| C1 Projection Profile | 각도 후보 스윕 횟수 (1차 61회 + 2차 41회 = 102) | X — 검출이 아니라 그리드서치라 문서 내용과 무관하게 거의 항상 102 |

**`00939648` 문서(2페이지) 테스트 결과 요약**
- A1/A2/C1은 서로 잘 수렴 (0~1.6° 범위), 육안상 더 기울어진 페이지일수록 세 방법 모두 값이 커짐 → 신뢰할 만함
- **B1(EasyOCR bounding box)만 계속 3° 근처로 높게 나옴.** median+MAD 필터든 최빈구간(dominant cluster) 방식이든 집계 방법을 바꿔도 결과가 거의 동일 → 집계 로직 문제가 아니라, EasyOCR이 검출한 개별 quad 각도 자체가 박스마다 0~4.5°로 넓게 흩어져 있기 때문으로 확인됨 (CRAFT 계열 검출기는 IoU 기준으로 학습돼서 회전각 정밀도까지는 보장하지 않는 것으로 추정). **이 문서 조건에서는 EasyOCR 박스 각도가 A1/A2/C1보다 신뢰도 낮은 신호.**
- 처리 시간(페이지당): A1/A2/C1은 1초 내외로 빠름. **B1(EasyOCR GPU 추론)이 1.5~2초로 제일 느림** — 위 "공짜" 가정과 반대.
- EasyOCR VRAM 실측: 모델 가중치 자체는 ~100MB(`torch.cuda.memory_allocated`)뿐이지만, `detect()` 실행 시 PyTorch 캐싱 할당자가 ~3.6GB를 reserve, nvidia-smi 기준 프로세스 전체는 baseline 대비 +4GB 정도 사용. RTX 4090(24GB)에선 병목 아니지만, 줄이려면 `canvas_size`를 기본값(2560)보다 낮춰서 호출 가능 (대신 작은 글자 검출력 저하 트레이드오프).

**시각 검증용 산출물**: 방법별로 `step_0_original_grid`(보정 전 원본+그리드) → `step_1~5`(방법별 검출/필터/각도표시/회전/재검증) → `step_6_before_after_grid`(step_0과 step_4를 나란히 hconcat, 보정 전후 한눈 비교)를 `mk_docu/images_step1_skew/method_.../<페이지>/`에 저장.

> **[2026-07-22 최종 추천 업데이트] C1 Projection Profile을 1순위로 추천.** 이유:
> 1. 애초에 이 섹션 위쪽에서 "스캐너로 스캔된 정형 문서(표 위주)에 특히 강함 — 우리 문서 조건과 잘 맞음"이라고 짚었던 방법인데, 실측 결과가 그 예상을 그대로 뒷받침함 (A1/A2와 각도가 잘 수렴, 페이지별 편차 없음)

> 4. 결정적으로, 원래 "B1은 이미 쓰는 EasyOCR 박스를 재활용해서 공짜로 얻을 수 있다"고 기대했던 부분이 실측에서 뒤집힘 — B1이 오히려 제일 느리고 제일 신뢰도 낮은 신호로 확인됨 (위 "구현 및 실측 결과" 참고)

## 1.2 원근 보정 (Perspective Correction)
모서리를 인식한 후 사각형으로 펼치는 작업.

### 수동 / 반자동 (사람이 개입)
- 사용자가 앱에서 4개 모서리 점을 직접 클릭해서 지정
- 이후 `cv2.getPerspectiveTransform()` + `cv2.warpPerspective()`로 변환

### 자동 처리
**자동 처리 시 문제점**
| 문제 상황 | 원인 |
|---|---|
| 배경과 문서의 대비가 낮을 때 | 흰 문서를 흰 책상 위에서 촬영 → 엣지 검출 실패 → 4각형을 못 찾음 |
| 문서가 접혀있거나 그림자가 있는 경우 | 그림자 엣지를 문서 경계로 오인식 |
| 다른 물체가 겹쳐있을 때 | 손가락, 다른 서류가 걸쳐 있으면 윤곽선이 문서 모양이 아니게 됨 |
| 문서 경계가 이미지 프레임 밖으로 나갈 때 | 4개 꼭짓점을 다 못 찾음 |
| 다중 문서 / 복잡한 배경 | 가장 큰 윤곽선이 문서가 아닐 수 있음 (책상 전체가 잡히는 등) |
| 곡률(휘어짐) | 책을 스캔한 경우 등은 단순 4점 원근 변환으로 부족, 별도의 warping 알고리즘 필요 |

### 딥러닝 기반 (곡률/왜곡 보정)
- **DocUNet** (CVPR 2018, Stony Brook & Megvii)
- **[2026-07-21 추가] DocTr / DocTr++** (Document Image Transformer, 2021~2022) — Transformer 기반으로 DocUNet보다 이후 나온 모델. 기하학적 왜곡(unwarping) + 조명(illumination) 보정을 함께 처리. DocUNet 대비 더 최신이고 벤치마크 성능도 더 좋음
- **[2026-07-21 추가] UVDoc** (2023, 합성 데이터 기반 unwarping) — 곡률 보정과 조명 보정을 동시에 처리하는 최신 모델. 다만 "책을 펼쳐서 촬영한" 시나리오에 최적화되어 있어, 우리처럼 평판 스캐너 위주 문서에는 과한 도구일 가능성 높음

> 실무에서는 **"자동 검출 + 사용자 확인/수정"** 하이브리드 방식이 많이 쓰임. 자동으로 4점을 추정한 뒤, 사용자에게 보여주고 필요시 드래그로 조정하게 하는 식 (모바일 스캐너 앱들의 표준 UX).
>
> **[2026-07-21 추가] 이 프로젝트 판단**: 위 딥러닝 모델들(DocUNet/DocTr/UVDoc)은 모두 "폰 카메라로 찍은 책/서류의 휘어짐"을 보정하는 게 주 목적. 우리 문서가 평판 스캐너 스캔본 위주라면 이 섹션 전체가 낮은 우선순위 — 검수자 화면에서 스캔 문서가 심하게 휘어진 샘플이 실제로 보이는지 먼저 확인하고, 없으면 이 섹션은 스킵해도 무방.

### 딥러닝 기반 대안 — 경계 검출 (Segmentation)
최근엔 세그멘테이션 모델(예: 문서 경계를 학습한 U-Net류)로 더 강건하게 검출하는 방식도 많이 씀. 전통적 엣지 기반보다 배경이 복잡해도 잘 동작.

**모델**
- HP의 DeepLabv3 기반 튜토리얼
- Document-Scanner (Saeed-Biabani)
- dhSegment

## 1.3 크롭 (Crop)
불필요한 여백, 배경, 스캐너 테두리 제거.

### 수동 (사람이 개입)
- 스캔 후 뷰어에서 사용자가 직접 영역을 드래그해 지정
- 정확하지만 대량 처리 시 비효율적 (문서 100장이면 100번 다 확인해야 함)

### 자동 처리
**일반적 방법**
1. 배경(흰색/검정 여백)과 콘텐츠 영역을 구분하는 임계값 처리
2. 콘텐츠가 있는 최소 바운딩 박스 계산 (`cv2.boundingRect()` 등)
3. 여백만큼 잘라내기

> 원근 보정에서 찾은 4개 꼭짓점을 그대로 크롭 경계로 사용하는 경우도 많음 (원근 보정과 크롭이 사실상 한 단계로 묶이는 경우가 흔함)

## 1.4 잡음 및 마스킹(마커) 영역 처리 — [2026-07-21 신규 추가]

우리 문서의 핵심 특징 중 하나: **개인정보를 수기로 마킹펜(형광펜/네임펜류)으로 긋고 스캔**한 경우가 섞여 있음. 이건 순수 기하학적 보정(디스큐/원근/크롭)과는 별개 문제라, 별도 단계로 다뤄야 함.

### 왜 문제가 되는가
| 문제 | 설명 |
|---|---|
| OCR 오인식 | 마커로 덮인 영역을 EasyOCR이 "텍스트 있음"으로 오판, ko-trocr가 마커 밑 글자를 억지로 "추측"해서 잘못된 문자를 뱉어낼 수 있음 (환각) |
| 이면 비침 (bleed-through) | 스캐너로 스캔 시 종이 뒷면 인쇄/마커가 비쳐 보이는 경우, 앞면 텍스트와 겹쳐 노이즈로 인식됨 |
| 스캔 노이즈 | 저가/구형 스캐너 특유의 salt-and-pepper 노이즈, 스캔 라인(줄무늬) |
| 마커 색상 다양성 | 검정/파랑/형광 등 마커 색이 문서마다 달라 일괄 임계값 처리가 어려움 |

### 권장 접근
1. **마스킹 영역 자체를 "정상 텍스트"로 취급하지 않고 별도 클래스로 분류**
   - 색상 기반(HSV) 또는 간단한 CNN 분류기로 "마커로 덮인 블록" 감지 → 해당 영역은 OCR 대상에서 제외하고 `masked_region`처럼 메타데이터로 표시 (본문 파싱 로직의 `needs_ocr_review`와 유사한 패턴 재사용 가능)
   - 실제로 이 영역은 "인식 실패"가 아니라 "의도된 마스킹"이므로, report에도 별도 상태로 남겨야 검수자가 "이건 원래 이렇게 마스킹된 것"이라고 빠르게 판단 가능
2. **이면 비침 제거**: `cv2.ximgproc` 계열 또는 단순 adaptive threshold로 흐릿한 뒷면 글자(대비가 낮음)와 진한 앞면 글자(대비가 높음)를 구분
3. **스캔 노이즈 제거**: `cv2.fastNlMeansDenoising` (Non-local Means) 또는 median blur — salt-and-pepper 노이즈에 효과적, 텍스트 획 손상 적음
4. **표준 파이프라인 순서 제안**: 디스큐 → 노이즈 제거 → 이진화/마스킹 영역 분리 → (필요시) OCR

> 검수 UX 관점: 마스킹 영역을 자동 감지해서 원본 이미지 위에 "여기는 마스킹됨" 표시를 미리 해두면, 검수자가 매번 마커 부분을 수동으로 판단할 필요가 없어짐. 검수가 최우선 과제라는 점을 고려하면 이 자동 분류가 우선순위 높은 개발 항목.
